# 豆仓界面设计系统

## 方向与气质

- 使用场景：咖啡爱好者在豆柜旁或冲煮前后，单手快速确认余量、查看资料并记录一杯。
- 核心感受：像一本放在深色木架上的烘焙记录册；温暖、克制、紧凑，不像通用后台表单。
- 领域语言：咖啡豆、烘焙曲线、牛皮纸袋、黄铜秤、杯测表、风味轮与豆仓标签。
- 色彩世界：深烘褐、焦糖、蜂蜜金、牛皮纸、奶油白、木炭灰；状态色仅用于未开封、饮用中和已喝完。
- 标志性组件：杯形快捷喝一杯按钮及克数角标；“豆仓选择托盘”作为所有选项与日期选择的统一浮层语言。
- 图片留存：OCR 后出现“咖啡袋 / 标签 / 不保存”的归档确认，不自动保存用户没有确认的照片。
- 图片来源：OCR 和编辑页手动添加图片都使用豆仓风格“拍摄 / 读取图片”面板，不使用 Android 默认 prompt。

## 视觉基础

- 主题：优先沿用 Dark Roast；其余主题只替换既有 token，不改变信息层级。
- 深度策略：以低对比边框和同色阶表面变化为主，仅使用轻微阴影；禁止混用夸张投影。
- 表面层级：页面背景 → 卡片/表单 → 详情主卡 → 选择托盘或日历。越高层仅略微变亮。
- 字体：豆名、详情标题使用现有衬线展示字体；正文使用系统中文字体；克重、日期等数据使用紧凑且等宽感明确的数字排版。
- 间距基准：4px。常用间距为 4、8、12、16、20、24px。
- 圆角：小控件 10–13px，卡片 16px，主详情卡与底部弹窗 20–26px。
- 触控：独立图标按钮的可点区域不得小于 40×40px；禁用态必须同时降低对比度并阻止操作。

## 信息与交互层级

### 查看优先，编辑次级

- 点击咖啡豆卡片或饮用记录，默认进入只读详情页。
- 详情页右上角使用统一铅笔图标进入编辑；不得让普通卡片点击直接进入表单。
- 查看态使用内容卡片、属性网格和时间线；编辑态使用 `<dialog>` 表单。
- Android 返回键先关闭最上层选择托盘/日历，再关闭编辑器或详情页，不丢失未保存内容。

### 紧凑豆仓卡片

- 阅读顺序：豆名 → 烘焙商/产地 → 状态与属性标签 → 烘焙日期/余量。
- 左侧 4px 状态色轨必须保留。
- 快捷喝一杯使用杯形图标和克数角标，不单独占用一行，也不能触发卡片详情点击。
- 列表以信息密度为优先，单卡目标高度约 112px，卡片间距 8px。

### 饮用评分

- 整体评分和高级维度均允许留空。
- 不显示额外的“清空评分”按钮；再次点击当前等级即取消评分。
- 正向维度使用笑脸，苦涩感使用哭脸；选中范围通过主题强调色表达。

### 豆仓选择托盘

- 产地、烘焙商、处理法等 Smart Select 必须继续保留 select、hidden input、自定义输入三件套和管理入口。
- 用户可见层不得调用 Android 原生下拉菜单；使用底部 `<dialog>` 托盘展示选项。
- 当前值使用主题强调边框、轻色背景和勾选标记。
- “自定义”选项返回表单后显示文本输入，历史值仍可在管理面板中批量改名或删除。
- 烘焙度、状态、冲煮方式与未来固定枚举均复用同一托盘。

### 日期与时间

- 不使用 Android 原生 date/datetime-local 面板。
- 日期选择器使用月份标题、前后月按钮、周一开头的 7 列日历以及明确的今日/选中态。
- 饮用时间在同一日历下提供小时和分钟输入；必填时间不可清空。
- 日期值继续以 ISO 本地格式保存：`YYYY-MM-DD` 或 `YYYY-MM-DDTHH:mm`。

### 搜索

- 顶部搜索是全局入口，豆仓页搜索豆名、产地、处理法和风味，饮用页搜索豆名、冲煮方式、备注、克数和饮用时间。
- 切换“豆仓 / 饮用”标签时只替换 placeholder 和结果语义，不隐藏搜索按钮。
- 搜索结果计数必须显示“显示 X / 共 Y”，避免用户误以为数据丢失。

### 图片留存

- 咖啡袋和标签图片属于咖啡豆详情的一部分，显示在详情属性之后、风味笔记之前。
- 图片卡片使用同一表面层级和 15px 圆角，图片以 `object-fit: cover` 裁切，角标标明“咖啡袋”或“标签”。
- OCR 识别图只在用户明确选择后归档到 App 私有空间；取消或返回时清理临时图，不写入系统相册。
- 非 OCR 场景也允许在编辑页主动添加或更换咖啡袋、标签图片；保存咖啡豆后才正式归档。
- 所有拍摄/读取图片入口必须先弹出产品内来源面板，保持与选择托盘一致的深色表面、焦糖边框和双选项卡片。
- 图片留存不新增相册或通用存储权限；仍保持本机离线隐私叙事。

## 一致性检查

- 新页面是否先提供查看态，而不是直接暴露编辑表单？
- 新枚举或日期字段是否复用了豆仓选择托盘或自绘日历？
- 新图片入口是否提供明确归档选择，并避免无确认保存？
- 图标是否用于明确动作，而非装饰？
- 所有间距是否落在 4px 基准上？
- 边框、表面和强调色是否来自现有主题 token？
- 是否覆盖默认、按下、焦点、选中、禁用、空数据和错误状态？
